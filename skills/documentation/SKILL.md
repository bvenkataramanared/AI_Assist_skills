# Documentation

> Status: scaffold — content to be implemented.

Produces accurate, useful documentation that matches the code as it actually is:
docstrings, READMEs, changelogs, API references, and decision records.

## When to use

- Code is done and needs documenting, or docs have drifted from the code.

## When NOT to use

- Understanding code for yourself → use `explain-code`.

## Workflow

_To be implemented._

1. Identify the audience and the doc type needed.
2. Read the code to document it accurately (never guess behavior).
3. Match the repo's existing doc style and location conventions.
4. Write concise, correct docs; keep examples runnable and current.

## Outputs

- Accurate docstrings / comments / README / CHANGELOG / ADR as requested
- Consistent with repo conventions
- Examples that match real behavior
